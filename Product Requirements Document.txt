Product Requirements Document
TB-FO Assistant: Digital TB-01 Record, FO Work Diary, DOT Tracking & TB Field Workflow System

Version: 1.0
Prepared for: Codex implementation
Primary user: BRAC TB Field Organiser / Field Officer
Working title: TB-FO Assistant
Alternative title: TB-01 Digital Record System
Language: Bangla-first, English-supported
Context: BRAC TB Control Programme-style field workflow for patient registration, diagnosis tracking, treatment/DOT follow-up, contact investigation, TPT, outcome, reporting, and automatic FO daily diary.

1. Product summary

TB-FO Assistant is an offline-capable web application for TB field staff to manage the complete TB patient workflow from presumptive identification to treatment outcome.

The current prototype already includes a strong single-file HTML/CSS/JavaScript implementation with:

TB patient list
TB-01 style patient form
DOTS/SS information
diagnosis and lab result section
treatment history
drug regimen table
daily DOT grid
sputum follow-up table
contact investigation table
treatment outcome section
summary dashboard
search/filter
single-record HTML download
all-record JSON export
localStorage persistence
Bangla UI

The next product should evolve this into a complete FO field management system with automation, task generation, audit logs, reports, offline backup, and a dedicated FO Daily Work Diary.

WHO’s 2024 TB Digital Adaptation Kit recommends that TB digital systems include user personas, workflows, core data elements, decision-support logic, indicators, and functional/non-functional requirements. This PRD follows that structure so the system can be built cleanly and later aligned with national TB programme forms and indicators.

2. Product vision
2.1 Vision statement

Build a Bangla-first digital field assistant that helps an FO track every TB patient, every sample, every DOT dose, every contact investigation, every TPT client, every follow-up, and every daily work activity from one simple offline-capable app.

2.2 Product tagline

“From TB case finding to treatment outcome — one FO dashboard.”

2.3 Core promise

The software should help the FO answer these questions instantly:

Who are my active TB patients?
Who missed DOT?
Whose sputum follow-up is due?
Which lab results are pending?
Which positive patients have not started treatment?
Which contact investigations are incomplete?
Which contacts need TPT follow-up?
Which patients are near treatment completion?
What work did I do today?
What monthly report do I need to submit?
3. Problem statement

FOs manage many activities manually: patient records, DOT follow-up, sputum follow-up, contact investigation, TPT tracking, lab coordination, private-provider referral, SS supervision, outcome tracking, and monthly reporting.

Manual systems create several risks:

Problem	Consequence
Records are paper-based or scattered	Hard to search, track, and report
DOT missed doses are not automatically flagged	Higher loss-to-follow-up risk
Lab results are not tracked from sample to result	Delayed diagnosis or treatment initiation
Contact investigation is hard to monitor	Household contacts may remain unscreened
TPT clients are easy to lose	Preventive treatment completion suffers
Monthly reporting is repetitive	FO spends time counting instead of doing fieldwork
Daily FO activity is not automatically recorded	Supervisor cannot easily verify field work
No centralized worklist	FO must remember many tasks manually

The product solves this by making the system task-driven, patient-centered, and diary-backed.

4. Goals and non-goals
4.1 Goals
Primary goals
Digitize TB-01 style patient records.
Track patient diagnosis, treatment, DOT, sputum follow-up, contact investigation, TPT, and outcome.
Automatically create daily FO work diary entries when work is done.
Generate FO daily worklist based on due/overdue tasks.
Generate summary dashboards and monthly reports.
Work offline on a field device.
Support Bangla UI with English clinical/programme terms.
Export and backup data safely.
Prepare the system for future backend sync, login, SMS, and NTP-compatible reports.
Secondary goals
Track SS/DOT provider performance.
Track private provider referrals.
Track lab sample movement.
Track field outreach and ACF activities.
Add risk scoring for patients likely to become LTFU.
Add supervisor dashboard.
Add role-based access.
4.2 Non-goals

The system should not:

Replace clinical diagnosis.
Automatically prescribe medication.
Automatically finalize treatment outcome without staff confirmation.
Replace official NTP/e-TB Manager systems unless formally integrated.
Store sensitive patient data in insecure storage for production use.
Make HIV-related clinical decisions automatically.
Send patient data to third-party services without explicit consent and security review.

Digital adherence tools such as SMS, medication event monitoring, and video-supported treatment can support TB treatment adherence, but WHO frames these as tools that require careful implementation, context review, and patient-centered planning.

7. Core workflows
7.1 Patient registration workflow
Trigger

FO clicks “নতুন নিবন্ধন”.

Steps
FO enters TR number if available.
FO enters patient name, age, sex, phone, address, ward, union, upazila, district.
FO enters DOTS centre, SS, DOT provider, referral source.
System checks duplicates.
FO saves record.
System creates diary entry: New Patient.
System creates default status: Pre-treatment unless treatment start date exists.
Patient appears in registry and dashboard.
Acceptance criteria
User cannot save a completely empty patient record.
User can save with either patient name or TR number.
Duplicate warning appears if same name + phone or same TR exists.
New patient appears at top of list.
Diary logs patient registration automatically.
7.2 Diagnosis and lab workflow
Trigger

FO receives or enters lab result.

Steps
FO enters test type: GeneXpert, Truenat, microscopy, X-ray, culture.
FO enters lab ID, test date, result.
System flags result category:
MTB detected
Rifampicin resistant
Not detected
Invalid/error
Pending
If positive result and no treatment start date, system creates Treatment Start Pending task.
If RR/Rif-resistant result, system creates Urgent DR-TB referral flag.
Diary logs Lab Updated.
Acceptance criteria
Result updates are saved.
Lab result change creates diary entry.
Positive result without treatment start appears in Today’s Worklist.
Invalid/error result creates retest task.
RR result shows high-priority red flag.
7.3 Treatment initiation workflow
Trigger

FO enters treatment start date.

Steps
FO selects patient type: new, relapse, treatment after failure, LTFU, transfer in, other.
FO enters treatment start date.
System calculates:
IP end date
treatment end date
follow-up sputum due dates
default DOT calendar months
System changes phase to Intensive Phase if appropriate.
FO assigns DOT provider.
Diary logs Record Updated.
Acceptance criteria
Start date automatically calculates IP end and treatment end.
DOT grid is generated for 6 months by default.
Follow-up sputum due dates appear.
Patient moves from Pre-treatment to active treatment.
7.4 DOT tracking workflow
Trigger

FO or DOT provider marks daily DOT.

Steps
User opens DOT grid.
User clicks a date cell.
Cell cycles through:
blank
Done
Missed
Supervised
System recalculates monthly adherence.
If missed, system creates follow-up task.
Diary logs DOT Updated.
Acceptance criteria
DOT status persists after save/reload.
Monthly adherence percentage updates instantly.
Missed DOT appears in Today’s Worklist.
Diary logs each DOT update with date and patient.
7.5 Sputum follow-up workflow
Trigger

Patient reaches 2M, 5M, or 6M follow-up.

Steps
System generates sputum due task.
FO enters sample/test date, lab ID, microscopy, GeneXpert/Truenat, culture, weight, comments.
System clears due task.
System flags positive follow-up result.
Diary logs Lab Updated or Sputum Updated.
Acceptance criteria
2M/5M/6M follow-up rows exist.
Due/overdue follow-up appears on dashboard.
Positive follow-up result creates alert for review.
Final negative follow-up can support outcome suggestion.
7.6 Contact investigation workflow

Bangladesh NTP publishes contact investigation and TPT-related forms, including Contact Investigation Form, TPT Register, and TPT Card, so this module should remain compatible with those concepts.

Trigger

Patient is bacteriologically confirmed pulmonary TB.

Steps
System auto-creates CI task.
FO enters CI date and investigator.
FO adds household/close contacts.
For each contact, FO enters:
name
age
sex
relationship
symptoms
referral status
investigation type
result
outcome
TR/TPT number
follow-up date
System flags child contacts and symptomatic contacts.
System suggests TPT review for eligible contacts.
Diary logs CI Updated.
Acceptance criteria
CI task appears for eligible index cases.
Contact rows can be added/edited/deleted.
Symptomatic contacts appear in referral task list.
TPT candidates appear in TPT module.
CI activity is logged in diary.
7.7 TPT workflow
Trigger

Contact is eligible for TB preventive treatment.

Steps
FO marks contact as TPT eligible.
System creates TPT record.
FO enters TPT regimen, start date, expected end date.
System tracks follow-up and completion.
Missed TPT follow-up appears in Today’s Worklist.
Diary logs TPT Updated.
Acceptance criteria
Contacts can become TPT records.
TPT expected completion date auto-calculates based on regimen.
TPT pending/completed/defaulted status is reportable.
TPT follow-up due list is visible.
7.8 Outcome workflow
Trigger

Patient completes treatment, dies, transfers, defaults, or fails.

Steps
FO selects treatment outcome:
Cured
Treatment Completed
Died
Treatment Failure
Lost to Follow-up
Transfer Out
Not Evaluated
FO enters outcome date, transfer location, sign-off officer, comments.
System updates patient phase/status.
Diary logs Outcome Updated.
Patient appears in outcome reports.
Acceptance criteria
Outcome can be selected and saved.
Outcome date is required before final closure.
Outcome change appears in diary.
Completed patients no longer appear in active treatment count.
7.9 FO Daily Work Diary workflow
Trigger

Any meaningful FO action.

Steps
FO performs action: save patient, update DOT, update lab, add CI row, select outcome, download report, delete record.
System automatically creates diary entry.
Diary entry includes:
timestamp
date
action type
patient ID
TR
patient name
details
user
FO opens diary page.
FO filters by date/type/search.
FO exports diary.
Acceptance criteria
Diary logs automatically without manual typing.
Diary can filter by date and type.
Diary shows daily summary.
Diary can export JSON/CSV.
Deleting a patient still keeps delete diary entry.

16. Diary and audit behavior
16.1 Diary vs audit trail

The app should maintain two concepts:

FO Diary

Human-readable activity timeline.

Example:

“DOT 13 May 2026 তারিখে আপডেট: Done / ওষুধ খেয়েছে”

Audit Trail

Structured system-level change record.

Example:

{
  "entity": "patient",
  "entityId": "TB123",
  "field": "xpertResult",
  "oldValue": "",
  "newValue": "T — MTB Detected",
  "changedAt": "2026-05-13T10:40:00Z"
}

MVP can keep only diary. Production should add audit trail.

22. Key product decisions
22.1 Use “Diary” as both productivity and accountability

The diary should not require the FO to manually write every action. The app should generate diary entries automatically.

Manual note-taking can be optional later, but automatic logging is the core feature.

22.2 Keep clinical decisions human-reviewed

The system can suggest:

treatment start pending
DR-TB referral needed
outcome pending
possible cured/completed
TPT candidate

But clinical/programme staff must confirm final decisions.

22.3 Make the first version offline-first

Because field conditions may have weak internet, the MVP should be local-first. Backend sync can be added later.

22.4 Use Bangla-first UX

The FO’s daily workflow should be Bangla-friendly. English programme terms can remain where standard:

DOT
GeneXpert
Truenat
TPT
CI
BC/CD
LTFU
Treatment Outcome
23. Final product definition
The app is successful when:
An FO can register and manage all TB patients digitally.
The FO can see today’s required work without manually checking every record.
DOT missed doses are visible immediately.
Lab pending and treatment-start pending cases are not forgotten.
Contact investigation and TPT are trackable.
Treatment outcomes are recorded cleanly.
The FO’s daily work is automatically logged.
Monthly reports can be generated from the data.
Records can be exported, backed up, and printed.
The system works offline on a normal laptop or mobile browser.
